1. บทนำและภาพรวมระบบ (System Overview)

ระบบ PhayuWittaya DJ ถูกพัฒนาขึ้นเพื่อใช้เป็นแพลตฟอร์มในการบริหารจัดการสื่อกลางประเภทเสียงและดนตรีสำหรับกิจกรรมภายในโรงเรียน โดยมีสถาปัตยกรรมการทำงานมุ่งเน้นการทำงานผสานกันแบบเวลาจริง (Real-time Synchronized System) ผู้ใช้งานสามารถถูกแบ่งออกเป็น 2 กลุ่มหลัก ได้แก่ กลุ่มนักเรียน (Clients/Students) ซึ่งสามารถเข้าถึงระบบเพื่อส่งคำขอเพลงผ่านการดึงข้อมูลจากแพลตฟอร์มวิดีโอ (YouTube) และกลุ่มผู้ดูแลระบบ (Manager/DJ) ซึ่งมีสิทธิขาดในการควบคุมคิวเพลง ปิด-เปิดระบบ ตั้งตารางเวลา และปรับระดับเสียงส่วนกลาง

คุณลักษณะหลักของระบบอาศัยการกระจายข่าวสาร (Pub/Sub Pattern) ผ่านทางฐานข้อมูลคลาวด์ เพื่อรับประกันว่าทุกฝั่งจะเห็นสถานะคิวเพลงที่เป็นชุดเดียวกันเสมอ ระบบนี้แก้ปัญหาความสับสนจากการมีผู้ยื่นความจำนงขอเพลงหลายคนพร้อมกัน และลดภาระของผู้จัดรายการด้วยระบบ Auto-DJ ที่สามารถทำงานชดเชยในช่วงเวลาปลอดคำขอได้อย่างมีประสิทธิภาพ

2. สถาปัตยกรรมและเทคโนโลยี (Architecture and Technology)

เพื่อให้มีความพร้อมต่อการให้บริการตามมาตรฐานอุตสาหกรรมซอฟต์แวร์ ระบบได้ถูกบูรณาการผ่านองค์ประกอบและเทคโนโลยีหลากหลายภาคส่วน (Tech Stack) โดยเน้นไปที่ความคล่องตัวในการพัฒนา (Agility) และประสิทธิภาพสูงสุดของการทำงาน (High Performance) ทางฝั่งผู้ขอใช้บริการ

2

2.1 เทคโนโลยีส่วนหน้า (Frontend Architecture)

ในส่วนการประมวลผลและการเรนเดอร์กราฟิกทางฝั่งเครื่องรับ (Client-side Rendering) ระบบถูกพัฒนาบนพื้นฐานของไลบรารี React เวอร์ชัน 19 ประกอบร่วมกับเครื่องมือบิลด์ (Build Tool) อย่าง Vite เพื่อย่นระยะเวลาการนำส่งไฟล์ไปยังเบราว์เซอร์

  • Styling & UI Components: การกำหนดรูปลักษณ์ทั้งหมดใช้ Tailwind CSS เพื่อให้ได้การออกแบบที่มีความยืดหยุ่น สร้างสุนทรียภาพทางการมองเห็นขั้นสูง ผ่านการใช้แบบจำลองทางกายภาพจำลอง (Skeuomorphism) ในบางจุด เช่น แถบเลื่อนปรับระดับเสียง (Volume Slider) ที่สะท้อนการตอบสนองสมจริง
  • Progressive Web App (PWA): เพื่ออุดช่องโหว่ทางด้านเครือข่ายอินเทอร์เน็ตที่คาดเดาไม่ได้ โครงสร้างสถาปัตยกรรมส่วนหน้าถูกลงทะเบียนซอร์สโค้ดให้ปฏิบัติการผ่าน Service Worker ทำให้โมดูลหลักต่างๆ ของแอปพลิเคชันยังคงตอบสนองได้แม้อยู่ในสถานะออฟไลน์ (Offline-capable)

2.2 เทคโนโลยีส่วนหลังและฐานข้อมูล (Backend and Database)

หัวใจหลักของ PhayuWittaya DJ คือการเชื่อมต่อไปยังบริการคลาวด์ Firebase Realtime Database ซึ่งทำหน้าที่เป็นศูนย์กลางเก็บข้อมูลเชิงสถานะ (State Machine) ที่ถูกประสานข้อมูลอยู่เสมอ ไม่จำกัดตรรกะว่าจะเป็นข้อมูลประเภทข้อความหรือคำสั่งควบคุม

เมื่อตัวแปรฐานข้อมูล (เช่น คิวเพลงปัจจุบัน currentSong หรือคำสั่งระดับเสียง volume) ได้รับการเขียนทับ โหนด (Node) จะกระจายการแจ้งเตือนพุชไปยังผู้ใช้ทั้งหมดทันทีด้วยความหน่วงเวลาต่ำกว่าระดับมิลลิวินาที

3

2.3 โครงสร้างการส่งมอบสื่อกระจายเสียง (Media Delivery)

เพื่อควบคุมกระแสเสียงที่จะถูกส่งออกไป โมดูลเครื่องเล่นถูกพัฒนาบน react-player แต่นำมาห่อหุ้มใน Custom Hook ที่ชื่อ usePlayer.js อีกหลายชั้น เครื่องมือนี้จัดการปัญหาการเชื่อมต่อซ้ำซ้อนกับทาง YouTube IFrame API รับฟังการประกาศสถานะผ่านอีเวนต์ (onStateChange) ในระดับโปรโตคอล เพื่อประมวลผลการจบแทร็ก (Song End) ส่งผลลัพธ์ย้อนกลับไปตัดคิวฐานข้อมูลอย่างอัตโนมัติ

3. ความท้าทายและการแก้ไขปัญหาเชิงเทคนิค (Technical Challenges)

ในการพัฒนาโปรเจกต์มัลติดมีเดียผ่านเว็บเบราว์เซอร์ มักเผชิญกับข้อจำกัดทางสภาพแวดล้อม ทีมพัฒนาได้ทำการแก้ไขเชิงกลยุทธ์ต่างๆ (Workarounds) ต่อไปนี้:

3.1 การรักษาลำดับการเล่นเสียงไม่ให้สะดุด (Seamless Audio Playback)

ปัญหา: โครงสร้างแอปพลิเคชันเดิมที่ใช้ไลบรารีการจัดเส้นทาง (React Router) เมื่อผู้ใช้งานเปลี่ยนหน้ากระดานควบคุม ทำให้ Component เล่นไฟล์เสียงหลักหลุดจาก DOM Tree ส่งผลให้เพลย์แบ็กถูกระงับทันที

วิธีแก้ไข: นำระบบ Router มาตรฐานออก และรื้อสถาปัตยกรรมมาเป็น Strict Single-Page Application อย่างแท้จริง โดยใช้สภาวะส่วนกลาง (Global State - viewMode) ประกอบกับสถาปัตยกรรม React Portal ดันโมดูล Audio Player ให้อยู่ระดับรากสูงสุด (Top-level) เหนือ Layout ทุกระดับ ด้วยวิธีนี้ผู้ดูแลระบบสามารถผจญภัยสลับหน้าต่างได้อิสระ โดยไม่สูญเสียอินสแตนซ์ของเพลง

4

3.2 การหลีกเลี่ยงข้อจำกัดการสืบค้นผ่าน Scraping Proxy

ปัญหา: API มาตรฐานของแพลตฟอร์มมีขีดจำกัดด้านโควตา (Rate Limits) หากมีนักเรียนกดค้นหาข้อมูลมากเกินไป จะเกิดกรณีก่อให้เกิดค่าใช้จ่ายส่วนเกิน (Quota Exceeded)

วิธีแก้ไข: นวัตกรรมเบื้องหลังโมดูล youtubeService.js ได้ใช้กลไก Reverse Engineering โดยดึงผ่านพร็อกซี่สกัดข้อมูลตัวแปร ytInitialData ที่ฝังมากับ HTML ดั้งเดิม เทคนิคแบบ Web Scraping ช่วยลดการพึ่งพิง API แบบมาตรฐาน ทำให้ระบบขยายตัวรองรับการหาชื่อเพลงของนักเรียนนับพันคนได้อย่างไม่จำกัด

3.3 การคัดกรองเนื้อหาและระบบ Auto-DJ

ปัญหา: อุปสรรคประการหนึ่งในศูนย์กลางสาธารณะคือ การส่งเนื้อหาที่มีระยะเวลายาวเกินควร หรือภาวะ Dead air เพลงจบไม่มีคนต่อ

วิธีแก้ไข: การป้องกันล่วงหน้าได้ถูกออกแบบในรูป DurationChecker.jsx ซึ่งเป็นโมดูลล่องหน (Invisible Component) จะโหลดตัวแปร IFrame เพื่องัดเอา Metadata ของความยาวคลิป ออกมาตรวจสอบกฎก่อนอนุญาตบันทึกคิว นอกจากนี้ตรรกะในฟังก์ชัน handleSongEnd จะคอยจับสัญญาณ หากพบว่าคิวว่างเปล่า ระบบ Auto-DJ จะหยิบก้อนเนื้อหา Lofi (Fallback Playlist) ยัดกะหล่ำใส่สายลำโพงโดยทันทีเพื่ออุดรอยรั่วแห่งความเงียบ

4. โครงสร้างไฟล์และแผนผังโปรเจกต์ (Project Documentation)

สถาปัตยกรรมโครงสร้างโปรเจกต์ถูกจัดกลุ่มแบบวิเคราะห์ตามโดเมนหน้าที่ (Domain-driven Design) ซึ่งส่งผลให้ง่ายต่อการซ่อมบำรุงในอนาคต (Scalability Matrix) ดังต่อไปนี้:

5. บทสรุปและทิศทางในอนาคต (Conclusion)

โปรเจกต์ PhayuWittaya DJ แสดงให้เห็นพอร์ตโฟลิโอที่มีความแม่นยำทางวิศวกรรมสถาปัตยกรรมแอปพลิเคชันอย่างครบวงจร (Full-cycle System Construction) ตั้งแต่การคราฟต์ภาพรวมดีไซน์เพื่อผู้ใช้ (UX) สู่การร้อยเรียงท่อน้ำเลี้ยงข้อมูล (Realtime Database Dataflow) การแก้ปัญหาข้อบกพร่องสภาวะแวดล้อมอย่างชาญฉลาด ซึ่งไม่เพียงแค่ทำให้ระบบตอบสนองตามความต้องการธุรกิจ (Business Requirements) อย่างเรียบง่าย แต่ยังพยายามเพิ่มคุณค่าแอบแฝงด้วยความรื่นรมย์ (Delightfulness)

ในอนาคต โครงสร้างแอปพลิเคชันพื้นฐานนี้สามารถถูกสานต่อและปรับเปลี่ยน (Refactoring & Evolution) สู่แพลตฟอร์มที่มีระบบยืนยันตนด้วย OAuth เพื่อให้กลุ่มผู้ใช้ออกขยายขนาดครอบคลุมมากยิ่งขึ้นตามความก้าวหน้าของเทคโนโลยีฐานข้อมูลบนคลาวด์คอมพิวเตอร์
